# OMS Safe Module Refactor

## 何時使用
遇到以下情況時使用：
- 大檔拆分
- page / logic / services 分層收斂
- 新資料源接入但舊接口還不能砍
- 舊頁面轉新頁面
- 將舊核心邏輯抽到 shared / services

## 核心原則
1. 這不是自由重構，是安全收斂
2. 先保留舊接口，再切新邏輯
3. 每一步都可回退、可驗證
4. 嚴格最小修改，不做順手大整理
5. 不可同時改 UI、資料流、邏輯層三個大面向

## 執行步驟
### Step 1：確認邊界
先回答三件事：
- 哪些檔案是入口
- 哪些檔案是被依賴方
- 哪些函式 / class / route 不能斷

### Step 2：畫出舊路徑
列出：
- 現在誰呼叫誰
- 哪個接口是外部依賴
- 哪個頁面還在直接用 legacy

### Step 3：設計新落點
決定這次只做一件事：
- 抽哪一段
- 新檔放哪裡
- 舊檔保留什麼橋接

### Step 4：局部抽離
實作方式：
- 先新增新模組
- 把可獨立段落抽出去
- 舊接口先保留
- 舊頁 / 舊 service 改成轉呼叫新模組

### Step 5：局部切換
只切這次任務涵蓋的入口。
不要順手把其他相鄰模組一起搬掉。

### Step 6：驗證
至少做：
- import smoke
- route / page smoke
- 被橋接的舊接口仍可呼叫
- `python dev_guard/run_local_guard.py`

### Step 7：記錄後續 cleanup
若 legacy 暫時保留：
- 記錄保留原因
- 記錄未來清理條件
- 不在這次任務硬砍

## 常用判斷
### 可以抽的
- 純邏輯函式
- 純查詢函式
- 純寫入函式
- 共用 view model
- 共用 formatter / mapper

### 先不要抽的
- 高耦合 UI 區塊
- 大量依賴 session state 的段落
- 還沒搞清楚上下游的 legacy 主流程

## 輸出格式
### 分析版
- 舊路徑
- 新路徑
- 保留橋接策略
- 影響檔案
- 驗證點

### 執行版
- 任務名稱
- 原則
- Step 1 / Step 2 / Step 3
- 檔案
- 修改
- 保留
- 驗證

## 禁止事項
- 不可整檔搬空再回補
- 不可未保留舊接口就直接切斷
- 不可一口氣拆太多檔
- 不可為了美觀而大改命名
